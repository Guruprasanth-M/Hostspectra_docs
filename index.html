Docker Sentinel

Real-time security monitoring platform for Linux servers

1. Introduction

Docker Sentinel is a self-hosted, real-time security monitoring platform for Linux servers. It runs entirely inside Docker containers, collecting host-level telemetry through the /proc filesystem, scoring it with machine learning anomaly detection, evaluating configurable policy rules, and executing automated responses when threats are detected.

Docker Sentinel collects CPU, memory, disk, network, process, and port data every 5 seconds. An Isolation Forest ML model scores each batch for anomalies. Configurable policy rules evaluate scores and trigger alerts. An action engine can automatically block IPs or kill processes. A webhook service notifies external systems. A web dashboard displays everything in real time.

2.2 Data Flow

The data flows through the system in a pipeline pattern. Each stage reads from an upstream Redis stream and writes results to a downstream Redis stream.

  1. Collectors read /proc/stat, /proc/meminfo, /proc/net/tcp, and process data via psutil. They publish raw events to topic-specific Redis streams and build a 5-second aggregated feature vector.
  2. ML Engine consumes feature vectors, runs them through a pre-trained Isolation Forest model, and publishes anomaly scores. The model retrains on a rolling 24-hour window every 6 hours.
  3. Policy Engine evaluates scored metrics against YAML-defined rules. Rules have conditions, severity levels, and cooldown periods. Violations become alerts, and some trigger action requests.
  4. Action Engine executes automated responses: IP blocking via iptables, process termination via signals. It enforces whitelists, rate limits, and supports dry-run mode.
  5. Webhook Service delivers alerts to external targets (Slack, Discord, generic HTTP). Payloads are HMAC-SHA256 signed. Failed deliveries retry with exponential backoff.
  6. API Server reads from all Redis streams and serves data to the dashboard via REST endpoints and a WebSocket connection.

2.3 Service Communication

All inter-service communication happens through Redis Streams. No service calls another service directly. This provides:

  • Decoupling. Services can be restarted independently.
  • Buffering. If a downstream service is slow, messages queue in Redis.
  • Replay. New consumers can read historical data from the streams.
  • Ordering. Redis Streams guarantee insertion-order delivery.

4.2 Policy Rules

File: config/policies.yml

Policies define conditions under which alerts fire and optional automated actions. Each rule evaluates incoming anomaly scores from the ML engine.

version: "0.1"

policies:
  - name: elevated_risk_monitor
    enabled: true
    conditions:
      score_above: 0.40
    action: alert_only
    severity: medium
    notify: false

  - name: brute_force_block
    enabled: true
    conditions:
      score_above: 0.65
      anomaly_type: brute_force
      repeated_within_seconds: 60
      min_occurrences: 3
    action: block_ip
    severity: high
    notify: true

  - name: port_scan_alert
    enabled: true
    conditions:
      score_above: 0.55
      anomaly_type: port_scan
    action: alert_only
    severity: medium
    notify: true

  - name: high_risk_alert
    enabled: true
    conditions:
      score_above: 0.65
    action: alert_only
    severity: high
    notify: true

  - name: critical_threat
    enabled: true
    conditions:
      score_above: 0.85
    action: block_ip
    severity: critical
    notify: true

  - name: ssh_anomaly
    enabled: true
    conditions:
      score_above: 0.50
      anomaly_type: ssh_failure
      repeated_within_seconds: 30
      min_occurrences: 3
    action: alert_only
    severity: medium
    notify: true

  - name: process_spike
    enabled: true
    conditions:
      score_above: 0.60
      anomaly_type: process_spike
    action: alert_only
    severity: medium
    notify: false
FieldTypeRequiredDescription
namestringyesUnique rule identifier
enabledbooleanyesWhether the rule is active
conditions.score_abovefloatyesMinimum anomaly score (0.0–1.0) to trigger
conditions.anomaly_typestringnoMatch specific anomaly type (brute_force, port_scan, ssh_failure, process_spike)
conditions.repeated_within_secondsintegernoTime window for occurrence counting
conditions.min_occurrencesintegernoRequired occurrences within the time window
severitystringyesmedium, high, or critical
actionstringyesResponse type: alert_only or block_ip
notifybooleanyesWhether to send webhooks for this alert

How rules are evaluated:

  1. The policy engine reads scored metrics from sentinel:scores.
  2. Score data is unwrapped: Redis stores scores nested as {"data": {actual payload}}. The engine extracts the inner object.
  3. Each enabled rule checks whether score_data["score"] exceeds score_above.
  4. If anomaly_type is specified, it must match the score's anomaly_type field.
  5. Repeat conditions track occurrences within a sliding window and require min_occurrences hits.
  6. When a rule fires, the policy engine publishes an alert to sentinel:alerts with the severity and action.
  7. If action is block_ip, a request is also published to sentinel:actions for the action engine.
  8. If notify is true, the webhook service picks up the alert and delivers it to configured targets.
The example file (config/policies.yml.example) ships with higher thresholds for production use. The live config uses lower thresholds (e.g., 0.40 for elevated risk) to catch more events during initial tuning.

5. Dashboard

5.1 Accessing the Dashboard

Open http://YOUR_SERVER_IP:8080 in any modern browser. The dashboard is a single-page application that works without authentication by default.

5.2 Pages

Overview

Primary monitoring view. Risk score (0-100), number of online services, active alerts, anomaly count. Service health grid with live status indicators. CPU, memory, and network utilization bars. Real-time CPU history and network traffic charts.

Metrics

Detailed system metrics: CPU percent, memory used, disk usage, system load. Four time-series charts (CPU, memory, network I/O, anomaly score). Raw metrics table with status badges.

Alerts

Alert timeline with filtering by severity (critical, high, medium), time range, and text search. Severity counts for each level. Chronological alert cards with source and timestamp.

Processes

Running process table. Sort by CPU, memory, PID, name, or connections. Adjustable row limit (25/50/100/200). Filter for flagged (risky) processes only. Columns: PID, Name, CPU%, Memory, Connections, Status, Risk.

Ports

Open ports and listening services. Columns: Port, Protocol, Connections/min, Unique IPs, State, Service, Risk.

Logs

Real-time log viewer. Filter by level and source. Terminal-style display with auto-scrolling.

Actions

Manual security actions: Block IP form and Kill Process form. Both require confirmation. Action history table showing past actions with status.

Settings

System info: version, service count, uptime, Redis status, ML status. API token management. Service versions table.

Diagnostics

Self-diagnostic health check page. Tests all CSS, JS, and HTML templates. Checks every API endpoint. Validates service health through the API. Tests WebSocket connectivity. Checks data pipeline flow. Shows errors with fix suggestions users can follow without SSH access.

5.3 Real-Time Updates

The dashboard maintains a WebSocket connection to the API server at /ws/live:

  • 1-second metric updates — CPU, memory, network stats update every second.
  • Live uptime counter — Client-side timer tracks session uptime.
  • Event counter — Shows events-per-second received from the server.
  • Instant alerts — Anomaly detections appear as notifications.
  • Connection status — Header shows LIVE/Connecting/Offline indicator.
  • Auto-reconnect — Exponential backoff from 1s to 30s.

7. Services

7.1 Collectors

Supervisor process that runs 4 sub-collectors as concurrent asyncio tasks. Reads host system metrics via mounted volumes and publishes to Redis streams. Collection cycle runs every 5 seconds.

Sub-collectors

ModuleSourceOutput
log_collector.py/host_logs/ (/var/log mounted read-only)SSH failures, sudo attempts, service restarts
network_collector.py/proc/net/tcp, /proc/net/tcp6TCP connections, unique IPs, port scan candidates
process_collector.pypsutil.process_iter()Process list, CPU spikes, memory spikes, new spawns
feature_builder.pyAggregates all sub-collector data14-field feature vector to sentinel:features

Collection Steps

  1. Read CPU usage from /proc/stat
  2. Read memory stats from /proc/meminfo and psutil
  3. Read disk usage via psutil.disk_usage()
  4. Read network connections from /proc/net/tcp
  5. Snapshot running processes via psutil.process_iter()
  6. Parse log files for security events (SSH, auth, sudo, service restarts)
  7. Build aggregated feature vector (14 fields)
  8. Publish raw metrics to sentinel:metrics and feature vector to sentinel:features

7.2 ML Engine

Scores metric batches using Isolation Forest (sklearn.ensemble.IsolationForest). 100 estimators, 0.05 contamination, StandardScaler normalization.

Scoring pipeline:

  1. Consume feature vectors from sentinel:features
  2. Normalize with fitted scaler
  3. Score with model.decision_function()
  4. Convert to anomaly probability (0.0 = normal, 1.0 = highly anomalous)
  5. Classify risk: normal (< 0.3), elevated (0.3-0.5), high (0.5-0.7), critical (> 0.7)
  6. Publish to sentinel:scores

Retrains every 6 hours on last 24 hours of data. A pre-trained model ships with the application for immediate anomaly detection.

7.3 Policy Engine

Evaluates scored metrics against configurable YAML rules. Reads from sentinel:scores, publishes alerts to sentinel:alerts.

Score Data Format

The ML engine publishes scores to Redis as {"data": "{json_string}"}. The policy engine parses each field value with json.loads(), then unwraps the outer "data" key to get the actual score payload:

# Score data arrives nested in Redis:
# {"data": '{"score": 0.43, "risk_level": "elevated", "anomaly_type": "normal", ...}'}

# After parsing and unwrapping:
# {"score": 0.43, "risk_level": "elevated", "anomaly_type": "normal", ...}

Consumer Group Pattern

All downstream services (policy, actions, webhooks) use Redis consumer groups. On startup, each service does a two-phase read:

  1. Phase 1 — Pending messages. Read with ID "0" to process any messages from a previous crash that were delivered but not acknowledged.
  2. Phase 2 — New messages. Once pending is drained, switch to ID ">" to read only new messages.
# Pending check: xreadgroup returns [[stream_name, []]] when PEL is empty
# This is truthy in Python, so we check for actual message content:
has_msgs = results and any(msgs for _, msgs in results)
if not has_msgs:
    processed_pending = True  # switch to ">"
The xreadgroup return format when reading pending messages with an empty PEL is [[stream_name, []]], which is a truthy list. Checking if not results alone would never transition to new messages, causing an infinite loop. Always check whether there are actual messages inside the response.

Rule Evaluation

For each scored metric:

  1. Parse and unwrap the score data.
  2. Iterate over all enabled rules from policies.yml.
  3. Check if score > rule.conditions.score_above.
  4. If the rule specifies anomaly_type, check the score's type matches.
  5. If the rule has repeat conditions, count occurrences within the sliding window.
  6. On match: publish an alert to sentinel:alerts, optionally publish to sentinel:actions.
  7. Acknowledge the message with XACK.

7.5 Webhook Service

Delivers alerts to Slack, Discord, and generic HTTP targets. HMAC-SHA256 signed payloads. Retry with exponential backoff (5s, 25s, 125s). Dead letter queue after 3 failures. Failed messages are stored in sentinel:webhooks:failed (max 1,000 entries) for manual inspection.

Uses the same consumer group pattern as policy and actions: two-phase read (pending first, then new), with proper empty-PEL detection.

Consumer Groups

Downstream services create consumer groups at startup with XGROUP CREATE. Each group tracks which messages have been delivered and acknowledged. This gives at-least-once delivery: if a service crashes mid-processing, the unacknowledged message stays in the group's pending entries list (PEL) and is re-delivered on restart.

# Check stream lengths
docker exec sentinel-redis redis-cli XLEN sentinel:metrics
docker exec sentinel-redis redis-cli XLEN sentinel:scores
docker exec sentinel-redis redis-cli XLEN sentinel:alerts

# Check consumer group info
docker exec sentinel-redis redis-cli XINFO GROUPS sentinel:scores

# Check pending entries for a consumer
docker exec sentinel-redis redis-cli XPENDING sentinel:scores policy_group

# Trim streams to prevent unbounded growth
docker exec sentinel-redis redis-cli XTRIM sentinel:metrics MAXLEN ~ 10000

Message Format

Each stream entry is a flat hash of string key-value pairs. The ML engine wraps its output in a "data" field containing a JSON string:

# sentinel:scores entry format:
{
  "data": "{\"score\": 0.43, \"risk_level\": \"elevated\", \"anomaly_type\": \"normal\",
             \"features\": {...}, \"timestamp\": \"2026-02-23T12:00:00Z\"}"
}

Downstream consumers must JSON-parse the "data" field value and then unwrap the outer object. This double-packaging is an artifact of how the ML engine serializes its output for Redis.

7.7 Process CPU Calculation

The API server reads per-process CPU usage from the host /proc filesystem (mounted as /host_proc inside the container). The calculation uses the kernel's accounting data rather than sampling, which gives accurate lifetime-average CPU percentages.

Algorithm

# 1. Read system uptime (seconds since boot)
system_uptime = float(open("/host_proc/uptime").read().split()[0])

# 2. Get kernel tick rate
clk_tck = os.sysconf("SC_CLK_TCK")  # usually 100

# 3. For each process, read /proc/[pid]/stat
#    Field 13 = utime (user ticks)
#    Field 14 = stime (kernel ticks)
#    Field 21 = starttime (ticks since boot when process started)

total_cpu_seconds = (utime + stime) / clk_tck
proc_uptime = system_uptime - (starttime / clk_tck)

# 4. Average CPU% over the process lifetime, divided by number of CPUs
cpu_pct = (total_cpu_seconds / proc_uptime) * 100.0 / num_cpus
cpu_pct = clamp(cpu_pct, 0.0, 100.0)

Why Not psutil?

The API container cannot use psutil.process_iter() because psutil reads the container's own /proc, which only contains container processes. By mounting the host's /proc at /host_proc (read-only), we can enumerate all host processes and calculate their CPU usage directly from kernel counters.

Previous versions used cumulative tick counts with an arbitrary scale factor, which caused long-running processes like dockerd to report 100% CPU even when actual usage was under 1%. The uptime-based formula gives accurate results by dividing total CPU time by the process's age.

7.8 Network I/O

Network throughput (bytes/sec in and out) is read from the host's /proc/net/dev, not from psutil. This is necessary because psutil inside the container measures only container-internal traffic on the Docker bridge network, not actual host network interfaces.

# Parse /host_proc/net/dev
# Skip the two header lines, then for each interface:
#   Column 0: interface name (e.g. "eth0:")
#   Column 1: receive bytes
#   Column 9: transmit bytes
# Sum across all non-loopback interfaces.

# Calculate rate:
elapsed = current_time - previous_time
bytes_in_per_sec = (current_rx - previous_rx) / elapsed
bytes_out_per_sec = (current_tx - previous_tx) / elapsed

Falls back to psutil.net_io_counters() if /host_proc/net/dev is not available (e.g., when running outside Docker for development).

9. Security Model

Container Isolation

All services run in unprivileged Docker containers. No container runs as root on the host. Capabilities are restricted to minimum required.

Network Security

  • Private Docker bridge network for all communication
  • Only port 8080 exposed to host
  • Redis internal-only
  • No outbound connections except webhooks

Authentication

  • Bearer token via X-Sentinel-Token header
  • Open by default (suitable for private networks)
  • WebSocket accepts optional token query parameter
  • Webhook payloads HMAC-SHA256 signed

Input Validation

  • Pydantic schemas for all API inputs
  • XSS: all user inputs escaped before rendering
  • Oversized payloads and malformed JSON rejected

Action Safety

  • Auto-expiring iptables rules
  • System PID protection
  • Rate limiting (5 actions/min)
  • Dry-run mode

12. Troubleshooting

Container won't start

docker compose logs <service-name>

Common causes: Redis not ready, port 8080 in use, insufficient memory (need 1 GB).

Dashboard shows "Offline"

  • Check API: docker compose ps api
  • Check nginx: docker compose logs dashboard
  • Verify WebSocket: curl -v http://localhost:8080/ws/live

No metrics data

  • Verify collectors: docker compose logs collectors
  • Check Redis: docker exec sentinel-redis redis-cli XLEN sentinel:metrics
  • Ensure /proc is mounted in docker-compose.yml

High CPU usage

ML model training uses CPU every 6 hours (normal). If a service is using high CPU continuously, check for tight loops in Redis stream consumers. The xreadgroup call should always include a block timeout (e.g., 5000ms) to avoid spinning.

Actions not executing

  • Check dry-run: ACTION_DRY_RUN=true in compose
  • Check logs: docker compose logs actions
  • Check whitelist: protected IPs (RFC 1918, localhost) and processes (systemd, sshd, dockerd) can never be acted on

Rate limited (429)

Default: 300 req/min/IP. Wait 60 seconds or modify RATE_LIMIT in services/api/middleware.py.

Process CPU shows 100% when actual usage is low

This happens when process CPU is calculated using cumulative tick counts with an arbitrary scale factor. Long-running processes like dockerd accumulate millions of ticks, so dividing by CPU count and multiplying by 0.01 still yields 100%.

Fix: The CPU calculation uses the uptime-based formula: (total_cpu_seconds / process_uptime) * 100 / num_cpus. This divides the process's total CPU time by the time it has been running, giving the correct lifetime average.

If you see this issue, rebuild the API container:

docker compose build --no-cache api
docker compose up -d api

Alerts page shows no alerts

Three common causes:

  1. No policies.yml. The policy engine needs config/policies.yml (not just the .example file). Without it, default rules may have thresholds too high for current scores.
  2. Score data nesting. Scores are stored in Redis as {"data": {actual_payload}}. If the policy engine reads the top-level score field, it gets None (0.0) and nothing triggers. The engine must unwrap the "data" key first.
  3. Consumer group stuck on pending. If the xreadgroup pending check uses if not results:, it never transitions to reading new messages because [[stream, []]] is truthy. The fix is has_msgs = results and any(msgs for _, msgs in results).

Quick fix procedure:

# 1. Verify policies.yml exists
ls -la config/policies.yml

# 2. Check if alerts are flowing
docker exec sentinel-redis redis-cli XLEN sentinel:alerts

# 3. Check consumer group status
docker exec sentinel-redis redis-cli XINFO GROUPS sentinel:scores

# 4. If consumer group is stale, recreate it
docker exec sentinel-redis redis-cli XGROUP DESTROY sentinel:scores policy_group
docker exec sentinel-redis redis-cli XGROUP CREATE sentinel:scores policy_group $ MKSTREAM

# 5. Restart policy engine
docker compose restart policy

Network data shows near-zero or wrong values

psutil.net_io_counters() inside a Docker container measures the container's virtual interface, not the host's physical interfaces. This shows near-zero traffic because the API container itself does very little network I/O.

Fix: Network I/O is read from /host_proc/net/dev (the host's /proc/net/dev mounted read-only). This file lists every host interface with cumulative byte counts. The code sums bytes across all non-loopback interfaces and computes a rate.

Verify the mount exists in docker-compose.yml:

api:
  volumes:
    - /proc:/host_proc:ro

Consumer group stale state

If a service was running with buggy code that never acknowledged messages, the consumer group's PEL (pending entries list) may contain thousands of stale entries. The service appears to start normally but never processes new messages because it's stuck re-reading the same pending entries.

# Check for large pending count
docker exec sentinel-redis redis-cli XPENDING sentinel:scores policy_group

# Nuclear option: destroy and recreate the group at the latest position
docker exec sentinel-redis redis-cli XGROUP DESTROY sentinel:scores policy_group
docker exec sentinel-redis redis-cli XGROUP CREATE sentinel:scores policy_group $ MKSTREAM

# Then restart the service
docker compose restart policy
Destroying a consumer group means any unprocessed messages in the PEL are lost. Using $ as the start ID skips all existing messages and reads only new ones. Use 0 instead if you want to reprocess historical data.
