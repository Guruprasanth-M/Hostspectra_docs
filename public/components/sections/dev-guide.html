Developer Document

A step-by-step guide to how HostSpectra was built — from an empty folder to a working SIEM.

Tech Stack

  • Language: Python 3.11
  • API Framework: FastAPI
  • Message Bus: Redis 7.2 (Streams)
  • Database: PostgreSQL 16
  • Dashboard: Nginx + vanilla JavaScript + echarts.js
  • ML: scikit-learn Isolation Forest
  • Orchestration: Docker Compose v3.11

Step 3 — Building The Collectors

The collectors are the foundation. Three parallel collectors run inside a single container:

  • Log Collector — Tails /var/log/auth.log, /var/log/syslog, parses SSH failures, sudo attempts, service restarts. Publishes to sentinel:logs Redis stream.
  • Network Collector — Reads /proc/net/tcp every 5 seconds, diffs snapshots, detects new connections and port scans (>5 ports from same IP in 30s). Publishes to sentinel:network.
  • Process Collector — Uses psutil to poll running processes, flags high CPU (>80%), high memory (>500MB), suspicious parent-child relationships (e.g., nginx spawning bash). Publishes to sentinel:processes.

A Feature Builder aggregates all three streams every 5 seconds into a 12-dimensional feature vector and publishes to sentinel:features.

Step 4 — ML Scoring Engine

The ML service reads feature vectors and calculates risk scores using a hybrid ensemble:

final_score = 0.5 × isolation_forest + 0.3 × z_score + 0.2 × ema_deviation
  • Isolation Forest — Pre-trained on baseline behavior. Points needing fewer partitions to isolate = more anomalous.
  • Z-score — Statistical deviation from running mean. Catches sudden spikes.
  • EMA deviation — Exponential moving average tracks gradual drift over time.

Scores (0.0–1.0) are published to sentinel:scores.

Step 5 — Policy Engine & Alerts

The policy engine reads scores and evaluates them against rules defined in policies.yml:

# Example policy rule
- name: high_anomaly_score
  condition: score > 0.6
  severity: high
  actions: [alert, block_ip]
  cooldown: 300

It deduplicates alerts (same type within 60s = single alert), persists to PostgreSQL, and publishes action requests for severe threats.

Step 6 — Actions Engine

The actions engine executes automated responses on the host:

  • IP Blocking — Adds iptables DROP rules via nsenter to access the host network namespace. Auto-reverses after configurable timeout (default 60 min).
  • Process Kill — Terminates suspicious processes detected by collectors.
  • Safety guardrails — Protected IPs (127.0.0.1, ::1), protected ports (22, 51820), max 5 actions per minute.

Step 7 — Webhooks & Notifications

Reads alerts and delivers them to configured endpoints:

  • Formats payloads for Slack, Discord, or generic HTTP POST
  • Signs all payloads with HMAC-SHA256 for verification
  • Retries with exponential backoff on failure
  • Dead-letters permanently failed messages

Step 8 — API & Dashboard

The API (FastAPI) exposes REST endpoints (/health, /metrics, /status, /alerts) and a WebSocket at /ws/live that streams events every 2 seconds to the dashboard.

The Dashboard is a pure frontend SPA served by Nginx. It proxies /api/* requests to FastAPI. Charts are rendered with echarts.js. No server-side rendering — everything runs in the browser.

Step 9 — The Installer (main.sh)

A single 1060-line bash script handles everything:

./main.sh start    # Install dependencies, build images, launch all 10 containers
./main.sh stop     # Graceful shutdown
./main.sh restart  # Restart with health check
./main.sh destroy  # Remove everything
./main.sh status   # Health report
./main.sh logs ml  # Tail specific service logs

It auto-detects the OS package manager (apt, yum, dnf, pacman, apk), installs Docker if missing, generates random secrets for the .env file, builds all 8 custom images, and starts the stack.

Step 10 — Testing

115 tests across three levels:

  • Unit tests — Individual functions, scoring logic, feature extraction
  • Integration tests — Redis stream flow, API endpoints, database queries
  • E2E tests — Full pipeline from simulated attack to alert delivery

Run with: docker compose -f docker-compose.test.yml up --abort-on-container-exit
